## 1. 일손은 무엇인가 (제품 정체성)

시니어(55~75세) 단순노무직 구직을 오프라인 매체(벼룩시장·교차로)에서 모바일/온라인으로 옮기는 위치 기반 매칭 서비스.

핵심 모델은 "앱을 켜서 찾는" 것이 아니라 "카카오톡으로 받는" 것. 구직자가 조건(직종·위치 등)을 등록해두면, 구인정보와 매칭해 카카오톡 알림으로 일자리를 전달한다.

60대 이상 사용자는 새 앱 설치 유입이 거의 안 될 것으로 보고, 대부분 능숙하게 쓰는 카카오톡을 주 채널로 삼는다. 따라서 앱 UI는 보조 수단이고, 진짜 접점은 카카오톡 알림이다.

구직자 관점 우선순위:
1. 집에서 멀지 않은 근무지
2. 적정 보상(대개 최저임금 부근)
3. 본인이 할 수 있는 일

계약은 용역회사와 맺지만 실제 일터는 현장. 한 용역회사가 여러 현장을 보유하므로, 구직자에게는 회사명보다 근무지(현장 위치)가 더 중요하다.

장기 비전은 다웰솔루션과 통합해 채용→배치→운영을 잇는 Total Workforce OS. (단, 이는 백로그. 지금은 채용 매칭 MVP 검증이 우선.)

## 2. 앱 구조 (3분할)

일손은 원래 세 개의 분리된 앱으로 설계됨. 슬랙에 배포 주소 3개 확인.

| 앱 | 대상 | 접속 default | 비고 |
|----|------|-------------|------|
| 구직자 | 시니어 구직자 | 모바일 | 핵심. 카톡 알림 착지점 |
| 구인기업 | 관리소장·FM사 등 | web | 공고 등록 |
| 관리자(Admin) | 운영팀 | web | 수동 매칭·알림 발송 |

라우팅(현재 점검한 단일 레포 기준):

```
시니어 구직자          기업                  관리자
/                     /employer/signup      /admin/match
/login                /employer/login       /admin/notify
/auth/callback        /employer/post        /admin/employers
/select-role          /employer/manage
/register
/jobs
/jobs/:id
/my
```

구직자 페이지는 `src/pages/worker/`에 모여 있음. 매칭은 자동이 아니라 관리자가 수동 개입하는 구조(`/admin/match`)로 보임. 운영자가 위치 기반 상위 후보를 확인·발송.

## 3. 구직자 흐름 화면별 현황 + 개선점

점검 순서: 랜딩 → 로그인 → 역할선택 → 가입(4단계) → 완료 → 일자리 목록

### 3-1. 랜딩 (`/`)
현재: 브랜드, "내 주변 일자리, 바로 알려드려요", 통계 카드(1,200+/8,500+, **하드코딩 더미**), 카카오톡 시작 버튼, 전화번호 시작, 기업 로그인, 하단 설명 문구.

개선 방향:
- 메시지·설명 과다 → 대폭 축소
- 텍스트 설명 대신 디자인으로 직관성 확보
- 핵심 메시지 2개만: 브랜드(일손) + '카톡으로 시작'
- 전제: 사용자는 지인 추천을 거쳐 진입 → 앱이 스스로를 길게 설명할 필요 없음
- '카톡으로 시작'이 default 경로 → 이 버튼이 압도적 주연
- 더미 통계는 신뢰도 역효과 → 제거 검토

### 3-2. 로그인 (`/login`)
현재: 첫 화면과 기능 중복(둘 다 '카톡 시작'). 브랜드가 로고→텍스트로 바뀜. 동의 문구 포함.

개선 방향:
- 존재 이유 불분명 → 첫 화면에 흡수 가능한지 검토
- 같은 행동('카톡 시작')을 두 화면에 나눠 둔 구조 → 첫 화면 버튼이 곧장 카카오 인증으로 가게
- 동의는 첫 화면 버튼 아래 문구로 법적 요건 충족
- 브랜드 표현 불일치(로고 vs 텍스트) 정리

### 3-3. 역할 선택 (`/select-role`)
현재: 구직자/구인 2지선다.

개선 방향:
- 구직·구인 진입점이 이미 분리됨(`/register` vs `/employer/signup`) → 이 페이지 명분 약함
- 구직자가 압도적 다수 → 2지선다를 동등하게 줄 필요 의문
- default는 구직자 → 구직자용 안내를 넣거나, 없으면 페이지 삭제도 옵션
- 카피 톤 밋밋 → 토스/애플처럼 사용자 관점·짧고 결과 중심으로

### 3-4. 가입 (`/register`, 4단계)
실제 수집 데이터:
- Step 1: 이름(카카오 자동입력) + 전화번호 + 출생연도
- Step 2: 주소(동네까지)
- Step 3: 희망 직종(복수) + 가능 시간대(복수)
- Step 4: 완료

"다음으로" 버튼은 필수값이 채워져야 활성화(Step 1은 이름+전화번호). 앞서 안 넘어간 건 전화번호 미입력 때문(버그 아님).

개선 방향:
- 카카오가 닉네임만 주므로(아래 4-2 참조) 전화·주소는 직접 받아야 함 → 가입은 없앨 게 아니라 매칭 핵심 데이터 수집 지점
- Step 3 제목 '희망 조건' 부적절(보상·거리 안 물음) → "어떤 일을 원하세요?" 류로
- 희망 직종 항목 중복 → 경비/주차관리, 청소/미화 2개로 단순화. 나머지는 후순위 또는 '기타'
  - **주의: 기업 공고 등록(`/employer/post`)의 직종 분류와 같은 목록이어야 매칭됨. 한쪽만 바꾸면 안 됨**
- 디자인 전면 개선 필요

### 3-5. 가입 완료
현재: 체크 아이콘 + "가입 완료!" + 일자리 보러가기.

개선 방향:
- 코드상 `justify-center`라 실기기에선 세로 중앙(개발자도구 높이 때문에 위로 치우쳐 보였을 뿐). 실측 재확인
- 디자인 개선 필요

### 3-6. 일자리 목록 (`/jobs`)
현재: 위치 미설정 경고, 거리 필터(1/3/5km), 일자리 카드(거리·직종·급여·근무요일·바로 지원하기).

평가: 회사명보다 일의 종류·거리·보상을 앞세운 구성은 구직자 우선순위와 일치. 단, 이 화면은 능동 탐색용이 아니라 카톡 알림 착지점이라는 관점에서 다듬어야 함.

## 4. 앞 개발자·PM이 남긴 방향 (슬랙 이력, 2~3월)

오늘 Brian이 화면 보며 느낀 것과 거의 동일한 방향이 이미 정리돼 인계됨. 즉 새로 발견한 게 아니라 합의됐던 미완 과제.

### 4-1. UI 방향 (60대 기준, PM 정리)
- 큰 글씨, 버튼 크고 명확, 쉬운 말
- 한 화면에 공고 1개씩(아래 스크롤보다 옆으로 넘기기)
- 우선순위 1번: 홈(랜딩) 제거 → 공고 리스트로 바로 진입, 카드 1개씩 + 옆으로 넘김
- 공고 카드 필수정보 순서: 근무지명(아파트/학교/시설) > 직종 > 급여 > 근무시간
- 지도는 보조로 축소

→ 오늘 본 화면엔 랜딩이 그대로 있음. 이 방향은 미완으로 인계된 것으로 보임.

### 4-2. 카카오 연동 제약 (중요)
- 카카오 로그인은 구현됨
- 단, 비즈니스 계정이 아니라 권한 부족 → 현재 카카오 사용자의 **닉네임만** 받아옴
- 이메일·휴대폰번호·실명을 가져오려면 **카카오 비즈니스 전환 필요**
- 카카오 메시지(알림): 템플릿 6종 등록 완료. 단 현재는 테스트용이라 본인에게만 발송 가능
- 카카오싱크 검수 신청 필요(MVP 확정 후)

### 4-3. 디자인 토큰 출처
- 브랜드 컬러 `#E85C1E` / `#C84010`
- 배경 크림톤 `#F5F0EB`
- 카카오 버튼 `#FEE500`
- 코드에 hex 직접 박힘 → CLAUDE.md 컨벤션(Tailwind 토큰)과 충돌. 디자인 정리 시 토큰화 대상

### 4-4. 지도·매칭
- 네이버 지도 API(표시·주소검색) + 카카오맵(길찾기)
- 반경: 3km까지 도보, 그 이상 차량 기준 최대 20km
- 단 문서는 Next.js 환경변수명 기준 → 현재 CRA 코드에서 실제 연결 상태는 확인 필요

## 5. 확인 필요 항목 (다음 작업 전 검증)

1. 지도 API(네이버/카카오)가 현재 CRA 코드에 실제 연결돼 동작하는지
2. 관리자 매칭(`/admin/match`)·알림(`/admin/notify`)이 진짜 카톡 발송까지 되는지, 화면만 있는지
3. 구직자 희망 직종 분류 vs 기업 공고 직종 분류가 동일한 목록인지
4. 슬랙 방향(랜딩 제거→공고 직행 등)이 코드에 일부라도 반영됐는지
5. 3개 앱(구직/구인/관리자)이 현재 단일 레포인지, 별도 배포인지
